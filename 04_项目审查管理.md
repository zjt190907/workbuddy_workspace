# 项目审查管理

> 基于 PROJECT_005 的 AI Agent 代码审查方法论，适配个人项目管理场景。
> 核心思路：**分层审查 + 优先级分类 + 度量追踪**，让每个项目可量化、可追踪、可持续改进。

---

## 一、审查分层架构

个人项目不需要企业级的 CI/CD 流水线，但分层审查的思路同样有效：

```
┌─────────────────────────────────────────────┐
│              Layer 4: 业务逻辑层              │
│  目标是否达成、功能是否可用、交互是否合理       │
├─────────────────────────────────────────────┤
│              Layer 3: 安全合规层              │
│  密钥管理、敏感信息、依赖漏洞、资源泄漏         │
├─────────────────────────────────────────────┤
│              Layer 2: 代码质量层              │
│  复杂度、重复代码、异常处理、测试覆盖           │
├─────────────────────────────────────────────┤
│              Layer 1: 基础规范层              │
│  命名、格式、README、目录结构、Git规范          │
└─────────────────────────────────────────────┘
```

### Layer 1: 基础规范检查（每次提交前）

| 规则ID | 检查项 | 严重度 | 说明 |
|--------|--------|--------|------|
| BAS-001 | README 完整性 | warning | 每个 PROJECT 必须有 README，含项目说明、使用方法 |
| BAS-002 | 目录命名规范 | error | 必须遵循 `PROJECT_序号_项目名` 格式 |
| BAS-003 | 文件命名规范 | warning | Python: snake_case; Java: PascalCase类 + camelCase方法 |
| BAS-004 | Git 提交规范 | warning | 提交信息格式: `[项目名] 动作: 描述` |
| BAS-005 | 依赖声明 | error | Python 项目必须有 requirements.txt 或 pyproject.toml |
| BAS-006 | 无冗余文件 | warning | 不应存在 .pyc、__pycache__、临时文件在版本控制中 |
| BAS-007 | 编码一致性 | info | UTF-8 编码，LF 行尾 |

### Layer 2: 代码质量检查（每周审查）

| 规则ID | 检查项 | 严重度 | 说明 |
|--------|--------|--------|------|
| QLT-001 | 单文件行数 | warning | 单文件 > 300 行建议拆分 |
| QLT-002 | 函数参数数量 | warning | 参数 > 5 个建议封装为对象/字典 |
| QLT-003 | 异常处理 | error | 不允许空 catch 或仅 print 的 catch |
| QLT-004 | 魔法值 | warning | 未解释的数字/字符串常量 |
| QLT-005 | 重复代码 | warning | 超过 6 行的重复代码块 |
| QLT-006 | 测试覆盖 | suggestion | 核心 logic 应有对应测试 |
| QLT-007 | 注释质量 | info | 关键决策点需要注释，无用注释应删除 |

### Layer 3: 安全合规检查（上线前/重要变更）

| 规则ID | 检查项 | 严重度 | 说明 |
|--------|--------|--------|------|
| SEC-001 | 硬编码密钥 | critical | 禁止在代码中硬编码密码/token/API Key |
| SEC-002 | 配置文件安全 | error | config.yaml 中不得包含明文密钥，应使用环境变量 |
| SEC-003 | 日志泄露 | error | 日志中不打印密码/token |
| SEC-004 | 依赖安全 | error | 定期检查依赖是否有已知漏洞 |
| SEC-005 | 输入校验 | warning | 对外部输入（API参数、文件内容）做基本校验 |
| SEC-006 | 资源泄漏 | warning | Python: with 语句管理文件/连接；关闭不用的连接 |

### Layer 4: 业务逻辑检查（里程碑节点）

| 规则ID | 检查项 | 严重度 | 说明 |
|--------|--------|--------|------|
| BIZ-001 | 功能完整性 | error | 核心功能链路是否可走通 |
| BIZ-002 | 边界条件 | warning | 空输入、超大数据、并发场景是否处理 |
| BIZ-003 | 幂等性 | warning | 重复执行是否产生副作用 |
| BIZ-004 | 降级策略 | suggestion | 核心功能失败时是否有备选方案 |
| BIZ-005 | 目标达成 | info | 项目是否达成最初设定的目标 |

## 二、优先级分类体系

借鉴 Project 5 的 P0-P4 分类，适配个人场景：

| 级别 | 标签 | 含义 | 处理策略 | 典型场景 |
|------|------|------|----------|----------|
| P0 | 🔴 Critical | 必须立即修复 | 阻止继续开发，当天解决 | 硬编码密钥、数据丢失风险、核心功能不可用 |
| P1 | 🟠 High | 强烈建议修复 | 下一个开发周期前解决 | 性能瓶颈、异常吞没、边界未处理 |
| P2 | 🟡 Medium | 建议修复 | 记录待办，不阻塞 | 魔法值、命名不规范、缺少注释 |
| P3 | 🔵 Low | 建议改进 |有空再说 | 代码风格优化、更好的实现方式 |
| P4 | ⚪ Info | 仅供参考 | 不需要行动 | 学习资源、架构演进方向 |

## 四、审查工作流

### 定期审查节奏

| 频率 | 审查层级 | 触发条件 | 执行方式 |
|------|----------|----------|----------|
| 每次提交 | Layer 1 | git commit/push | 自查 + .gitignore |
| 每周 | Layer 1+2 | 周度回顾 | 运行审查脚本 + 人工复盘 |
| 上线前 | Layer 1+2+3 | 功能完成/里程碑 | 全量审查 |
| 里程碑 | Layer 1-4 | 项目目标达成检查 | 全面评估 |

### 审查执行模板

每次审查后更新仪表盘，格式如下：

```markdown
### 审查记录: [日期] PROJECT_[序号]_[项目名]

**审查层级**: Layer X
**发现问题**: [列表]
**优先级分类**: [P0-P4]
**改进计划**: [具体措施]
**下次审查时间**: [日期]
```

## 五、度量指标

借鉴 Project 5 的指标体系，适配个人场景：

| 指标类别 | 指标名 | 计算方式 | 目标值 |
|----------|--------|----------|--------|
| 质量 | 规范达标率 | Layer 1 通过项 / 总项数 | > 90% |
| 质量 | P0问题数 | Critical 问题总数 | = 0 |
| 质量 | 安全合规率 | Layer 3 通过项 / 总项数 | > 80% |
| 效率 | 审查覆盖率 | 已审查项目 / 总项目数 | = 100% |
| 效率 | 问题修复周期 | 发现→修复的平均天数 | P0 < 1天, P1 < 7天 |
| 成长 | 复利转化率 | 踩坑日志→改进措施的转化数 | > 80% |
| 成长 | 重复犯错率 | 同类问题再次出现的比例 | < 10% |
